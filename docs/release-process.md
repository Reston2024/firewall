# Release Process — v2.0 Local AI SOC

**Date:** 2026-04-08
**Tools:** Syft (SBOM), Grype (vuln scan), cosign v3 (signing)

## Release Checklist

### Pre-Release

1. All validation suites pass:
   ```bash
   bash scripts/validate-phase9.sh --full   # Malcolm health
   bash scripts/validate-phase10.sh          # Data ingestion
   bash scripts/validate-phase12.sh          # RAG corpus
   bash scripts/validate-phase13.sh          # SOC integration
   ```

2. No outstanding FAIL results across any validation script

3. Git working tree is clean: `git status` shows no uncommitted changes

### Tag Release

```bash
git tag -a v2.0.0 -m "v2.0 Local AI SOC — Malcolm NSM + Foundation-Sec-8B + RAG + SOC Integration"
git push origin v2.0.0
```

### Generate SBOM + Sign

```bash
bash scripts/generate-sbom.sh v2.0.0
```

This produces in `releases/v2.0.0/`:
- `sbom-repo.json` — CycloneDX SBOM of the git repository (SCA-01)
- `sbom-system.json` — CycloneDX SBOM of the deployed system (SCA-02)
- `grype-repo.txt` — Vulnerability scan of repo SBOM (SCA-03)
- `grype-system.txt` — Vulnerability scan of system SBOM (SCA-03)
- `v2.0.0-bundle.tar.gz` — All artifacts bundled
- `v2.0.0-bundle.cosign` — cosign signature bundle (SCA-04)

### Verify Signature

```bash
cosign verify-blob --bundle releases/v2.0.0/v2.0.0-bundle.cosign releases/v2.0.0/v2.0.0-bundle.tar.gz
```

### Post-Release

1. Update README.md with v2.0 shipped date
2. Run `/gsd:complete-milestone` to archive v2.0 phases

## SBOM Scope Limitations

Per the security sweep (E7-01), the repo SBOM generated by Syft has known limitations for shell-script-heavy repositories:
- Syft sees no package manifests in the firewall repo (shell scripts, not Python packages)
- The repo SBOM captures file metadata but not dependency trees
- The system SBOM (from supportTAK-server `/`) captures installed packages, Docker images, and Python venvs
- Two SBOMs together provide coverage: repo artifacts + deployed runtime

## Tool Installation

Install on the host that runs `generate-sbom.sh`. For v2.0 this is the
supportTAK-server (`opsadmin@192.168.1.22`): the repo is synced there
transiently before the release run, and all three tools live in
`/usr/local/bin`.

```bash
# Syft (SBOM generator)
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh \
  | sudo sh -s -- -b /usr/local/bin

# Grype (vulnerability scanner)
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh \
  | sudo sh -s -- -b /usr/local/bin

# cosign v3 (signing) — binary download pinned to latest v3.x release.
# v3 is REQUIRED because the --bundle flag used by generate-sbom.sh is
# mandatory in v3 and not supported by v2.x. generate-sbom.sh asserts
# `cosign version` reports v3.x and exits with a clear error otherwise.
LATEST=$(curl -s https://api.github.com/repos/sigstore/cosign/releases/latest \
  | grep '"tag_name"' | head -1 | cut -d'"' -f4)
curl -sLO "https://github.com/sigstore/cosign/releases/download/${LATEST}/cosign-linux-amd64"
sudo mv cosign-linux-amd64 /usr/local/bin/cosign
sudo chmod +x /usr/local/bin/cosign
cosign version   # must print GitVersion: v3.x.x
```

## Where to Run the Release Script

`generate-sbom.sh` is designed to run on a host that has (a) a current
checkout of the firewall repo and (b) all three tools in PATH. For v2.0
the release run happens on supportTAK-server via:

```bash
# Sync the repo to supportTAK (transient, outside any git tree there):
tar --exclude='.git' --exclude='releases' -czf - -C "$(pwd)" . \
  | ssh opsadmin@192.168.1.22 'mkdir -p /tmp/firewall-release && tar -xzf - -C /tmp/firewall-release'

# Run the release script on supportTAK:
ssh opsadmin@192.168.1.22 'cd /tmp/firewall-release && bash scripts/generate-sbom.sh v2.0.0'

# Pull the signed artifacts back to the laptop repo:
scp -r opsadmin@192.168.1.22:/tmp/firewall-release/releases/v2.0.0 ./releases/

# Clean up the transient remote copy:
ssh opsadmin@192.168.1.22 'rm -rf /tmp/firewall-release'
```

The `sbom-system.json` step inside `generate-sbom.sh` already uses
`ssh ${SSH_TARGET}` to scan the live system; running the outer script on
supportTAK means that self-loopback SSH call becomes a no-op from the
same host (the command still works since `ssh opsadmin@192.168.1.22`
from supportTAK itself resolves to localhost authentication).
