## CRITICAL WARNINGS — Read Before Starting

- **WUI-managed:** Suricata is managed via WUI. Do NOT edit `/var/ipfire/suricata/suricata-used-rulefiles.yaml` directly — the file header warns it is autogenerated and any changes will be overwritten.
- **emerging-policy.rules MUST NOT be enabled** in initial deployment — this ruleset blocks Linux package managers (APT, YUM patterns) and can disconnect SSH management sessions. It is explicitly a "never enable without extensive prior tuning" category.
- **ALWAYS enable Surveillance (monitor) mode first.** NEVER use Drop mode on day one. Monitor for 24-48 hours minimum before considering Drop mode.
- **suricata.yaml is overwritten by Core Updates.** Every IPFire Core Update regenerates `/etc/suricata/suricata.yaml` from IPFire's internal template, wiping manual customizations (EVE JSON settings, memcap values). A sha256 baseline is mandatory after every manual yaml edit.
- **EVE alerts are NOT in syslog.** Only Suricata engine start/stop events go to syslog. All alert data is only in `/var/log/suricata/eve.json`. Do not configure Phase 5 telemetry to read syslog for IDS alerts.
- **IPFire DBL is beta in CU200.** Do NOT enable IPFire DBL until it reaches stable promotion. Premature activation may generate false positives against legitimate traffic.

## Section 2: Enable Suricata via WUI (IDS-01, IDS-02, IDS-04)

**WUI path:** Firewall > Intrusion Prevention System

Apply ALL settings in this order BEFORE clicking Apply:

### Step 2.1: Enable IPS
Check the **"Enable IPS"** checkbox.

### Step 2.2: Zone Selection
Check both **RED** and **GREEN** zone checkboxes.
- Do NOT check BLUE or ORANGE unless those zones are actively routing traffic.
- Note: IPFire lead developer recommends enabling all active zones. IPFire deduplicates traffic internally — no double-processing penalty for selecting both zones.

### Step 2.3: Mode
Select **"Surveillance"** (monitor-only mode).
- This logs alerts but takes NO blocking action.
- NEVER select "Drop" on day one — wait for 24-48 hours of baseline monitoring first.

### Step 2.4: Ruleset Selection
In the Rulesets section:
- Enable the **"Emerging Threats Community"** checkbox.
- This is the ET Community free ruleset — no registration required, daily updates, broad coverage of scanning, C2, and malware.

### Step 2.5: Automatic Updates
Enable the **"Automatic Updates"** toggle for ET Community.
- Updates run daily via fcron.daily at 01:25 AM.

### Step 2.6: DO NOT enable IPFire DBL
IPFire DBL is in beta as of CU200. Leave unchecked.

### Step 2.7: DO NOT enable emerging-policy category
If per-category selection is shown, ensure **emerging-policy** is NOT checked. This category blocks Linux package managers and can cause self-lockout.

### Step 2.8: Click Apply
Click the **Apply** button. Suricata will start and rules will download.
- First download may take 1-2 minutes (~20,742 rules in the ET Community full set).
- The WUI may show a spinner or brief page reload during rule reload.

**Verify via SSH after Apply:**
```bash
# Verify Suricata process started
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'pgrep suricata && echo RUNNING'

# Check rule load count (expect 10,000+ rules)
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'grep "rules loaded" /var/log/suricata/suricata.log | tail -1'
```

Expected output: A line containing a rule count of 10,000 or more.

**Note on flowbit warnings:** After enabling ET Community rules, `suricata.log` will fill with lines like `"flowbit 'XXX' is checked but not set"`. These are expected and harmless — ET Community ships some rules in commented-out state, and their dependent flowbit checks produce these warnings. Suricata still loads and runs correctly. Confirm with: `grep "0 rules failed" /var/log/suricata/suricata.log`

## Section 4: Apply N100 Memcap Values (IDS-05)

**Context:** Default suricata.yaml has conservative defaults designed for low-end hardware. Apply tuned values appropriate for the N100 16GB single-channel DDR5 system. These values double or quadruple the defaults while remaining conservative relative to available RAM.

**Reference file:** `C:\Users\ablan\Firewall\configs\suricata\suricata-yaml-memcap-reference.yaml`

### Step 4.1: Back up suricata.yaml before editing
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'cp /etc/suricata/suricata.yaml /etc/suricata/suricata.yaml.bak.$(date +%Y%m%d)'
```

### Step 4.2: Apply memcap values via SSH

Apply each value with a targeted sed command. Verify each change after applying.

**a) stream.memcap: 256mb**
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 "sed -i '/^stream:/,/^[^ ]/{s/memcap: 64mb/memcap: 256mb/}' /etc/suricata/suricata.yaml"
```

**b) stream.reassembly.memcap: 512mb**
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 "sed -i '/reassembly:/,/^[^ ]/{s/memcap: 256mb/memcap: 512mb/}' /etc/suricata/suricata.yaml"
```

**c) stream.reassembly.depth: 1mb**
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 "sed -i '/reassembly:/,/^[^ ]/{s/depth: 1mb/depth: 1mb/; t; s/depth: [0-9]*[mk]b/depth: 1mb/}' /etc/suricata/suricata.yaml"
```

**d) stream.checksum-validation: no** (REQUIRED for Intel i226-V NIC hardware checksum offload)
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 "sed -i '/^stream:/,/^[^ ]/{s/checksum-validation: yes/checksum-validation: no/}' /etc/suricata/suricata.yaml"
```
Note: The i226-V NIC performs checksum calculation in hardware. Suricata sees incomplete checksums before the NIC finalizes them, generating floods of false ICMP invalid checksum alerts. Setting `no` suppresses these false positives.

**e) flow.memcap: 128mb**
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 "sed -i '/^flow:/,/^[^ ]/{s/memcap: 32mb/memcap: 128mb/}' /etc/suricata/suricata.yaml"
```

**f) flow.prealloc: 30000**
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 "sed -i '/^flow:/,/^[^ ]/{s/prealloc: 10000/prealloc: 30000/}' /etc/suricata/suricata.yaml"
```

**g) defrag.memcap: 64mb**
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 "sed -i '/^defrag:/,/^[^ ]/{s/memcap: 32mb/memcap: 64mb/}' /etc/suricata/suricata.yaml"
```

**IMPORTANT:** The sed commands above are fragile if the default values in the IPFire CU200 template differ from those shown. After each sed command, verify the change took effect with the grep in Step 4.3. If a sed command did not change the value (because the default is different), edit the relevant line directly:
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'vi /etc/suricata/suricata.yaml'
```

### Step 4.3: Verify all values applied
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'grep -E "memcap:|prealloc:|depth:|checksum-validation" /etc/suricata/suricata.yaml'
```
Compare output against values in `configs/suricata/suricata-yaml-memcap-reference.yaml`.

### Step 4.4: Restart Suricata to apply yaml changes
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 '/etc/init.d/suricata restart'
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'pgrep suricata && echo RUNNING'
```

**CRITICAL:** After applying memcap values, proceed immediately to Section 5 (sha256 baseline) before any Core Update.

## Section 5: Capture sha256 Baseline (IDS-08)

**Context:** IPFire Core Updates regenerate `/etc/suricata/suricata.yaml` from a template, wiping all manual customizations. The sha256 baseline detects this so you know when to re-apply settings. Run this step immediately after any manual suricata.yaml edit.

### Step 5.1: Deploy check-suricata-integrity.sh to IPFire
```bash
# From dev machine (Windows)
scp C:\Users\ablan\Firewall\scripts\check-suricata-integrity.sh root@192.168.1.1:/usr/local/bin/check-suricata-integrity.sh

# Fix CRLF and make executable
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'sed -i "s/\r$//" /usr/local/bin/check-suricata-integrity.sh && chmod +x /usr/local/bin/check-suricata-integrity.sh'
```

### Step 5.2: Create initial baseline
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'sha256sum /etc/suricata/suricata.yaml > /root/suricata-yaml.sha256 && cat /root/suricata-yaml.sha256'
```
Expected: A sha256 hash followed by the path to suricata.yaml.

### Step 5.3: Verify integrity script works
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'bash /usr/local/bin/check-suricata-integrity.sh'
```
Expected: All checks PASS.

### Step 5.4: Add baseline to backup include list
```bash
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'grep -q suricata-yaml.sha256 /var/ipfire/backup/include.user || echo "/root/suricata-yaml.sha256" >> /var/ipfire/backup/include.user'
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'grep -q "/etc/suricata/suricata.yaml" /var/ipfire/backup/include.user || echo "/etc/suricata/suricata.yaml" >> /var/ipfire/backup/include.user'
ssh -i ~/.ssh/ipfire_ed25519 root@192.168.1.1 'cat /var/ipfire/backup/include.user'
```

**After every Core Update:**
1. Run: `bash /usr/local/bin/check-suricata-integrity.sh`
2. If WARN (hash mismatch): re-apply memcap values from Section 4 and EVE JSON settings from Section 3
3. Update baseline: `sha256sum /etc/suricata/suricata.yaml > /root/suricata-yaml.sha256`

- [ ] check-suricata-integrity.sh deployed to /usr/local/bin/ and executable
- [ ] sha256 baseline created at /root/suricata-yaml.sha256
- [ ] check-suricata-integrity.sh returns all PASS
- [ ] /root/suricata-yaml.sha256 added to backup include.user
