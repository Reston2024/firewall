# ADR-0010: ECDSA-Only WUI Certificate

- **Date:** 2026-03-25
- **Status:** Accepted

## Context

IPFire's WUI runs on HTTPS at port 444. The management interface requires a TLS certificate for secure access. Investigation into IPFire's certificate generation revealed that IPFire generates ECDSA-only certificates — there is no RSA option available in IPFire's certificate generation tooling. The project's security policy (D-04) prohibits storing certificates or private keys in the git repository.

## Decision

Document the ECDSA-only WUI certificate as-is. Do not attempt to replace it with a custom certificate. The certificate is NOT stored in the git repository.

## Rationale

- IPFire's certificate generation is ECDSA-only — this is a platform constraint, not a configurable choice
- The self-signed certificate is acceptable for a management interface accessed only from the trusted LAN (GREEN zone)
- Storing certificates in git is a security anti-pattern (D-04) — private keys must not be committed to version control
- Replacing the certificate with a custom one would require maintaining a separate CA or accepting browser warnings for a custom self-signed cert — no benefit over the IPFire-generated cert
- ECDSA certificates are considered more secure than RSA equivalents at equivalent key sizes

## Consequences

- WUI certificate behavior and fingerprint are documented in `docs/wui-certificate.md` for reference during rebuild verification
- The rebuild script Section 0 prerequisites list cert provisioning as a manual step — the certificate is generated by IPFire on first boot and must be re-verified after rebuild
- No automated cert deployment — cert management is handled entirely by IPFire's internal tooling
- Browser will show a certificate warning for the self-signed cert — this is expected and documented in operational procedures
