**ADR-0005-ipfire-as-base-os.md:**
- Context: Needed a firewall OS for Intel N100 6-NIC mini-PC. Options were pfSense, OPNsense, IPFire, or bare Linux with iptables/nftables.
- Decision: Use IPFire 2.29 (Core Update 200) as the base operating system.
- Rationale: Purpose-built firewall distro with zone-based architecture (GREEN/RED/BLUE/ORANGE), built-in WUI management at :444, Pakfire package system, native Suricata IDS since CU131, Unbound DNS with DNSSEC, designed for appliance use cases. Already installed on hardware.
- Consequences: Limited to 4 named zones (RED/GREEN/BLUE/ORANGE) — extra NICs use bridge mode. No Docker support (rejected by IPFire developers). Telemetry stack must run off-box. All customization must work within IPFire's SysVinit and custom buildroot environment.

**ADR-0006-guardian-over-fail2ban.md:**
- Context: Need SSH and WUI brute-force protection. fail2ban is the standard Linux choice but is not available in IPFire's Pakfire package manager.
- Decision: Use Guardian (Pakfire add-on) for SSH and WUI brute-force protection.
- Rationale: Guardian is IPFire's native brute-force protection tool, available via Pakfire, managed through WUI, actively maintained (updated February 2025). fail2ban is not in Pakfire and installing it outside Pakfire violates the project constraint of IPFire-native tools only.
- Consequences: Brute-force protection managed via WUI (not CLI config files). Guardian protects SSH and WUI ports. Configuration is WUI-managed (not version-controlled in repo — documented in rollback README instead).

**ADR-0007-suricata-ids-monitor-first.md:**
- Context: Suricata IDS/IPS can run in monitor-only mode (logs alerts, takes no action) or active IPS mode (drops matching packets). Active mode risks blocking legitimate traffic if rules are not tuned.
- Decision: Deploy Suricata in monitor-only mode first, switch to active IPS only after tuning is complete.
- Rationale: Monitor mode allows baselining legitimate traffic patterns without lockout risk. False positives can be identified and whitelisted before enabling packet drops. Prevents self-lockout during initial deployment. ET Community ruleset needs tuning for home/SOHO traffic patterns.
- Consequences: Initial deployment only logs alerts — no active blocking. Must manually switch to active IPS via WUI after tuning period. Rule categories enabled incrementally (IDS-07) to prevent overwhelming the system.

**ADR-0008-dns-over-tls-unbound.md:**
- Context: DNS queries to upstream resolvers travel in plaintext by default, exposing browsing data to ISP and network observers. IPFire uses Unbound as its DNS resolver with DNSSEC support.
- Decision: Enforce DNS-over-TLS (DoT) for all upstream DNS queries using Cloudflare (1.1.1.1) and Quad9 (9.9.9.9) as upstream resolvers. Disable ISP DNS.
- Rationale: DoT encrypts DNS queries to upstream resolvers (port 853). DNSSEC validates response integrity. Disabling ISP DNS is required — ISP DNS and TLS protocol are mutually exclusive in IPFire's configuration. Cloudflare and Quad9 both support DoT and have strong privacy policies.
- Consequences: ISP DNS must be disabled first (mutual exclusivity constraint in IPFire). All upstream DNS traffic uses port 853 (verifiable via tcpdump). DNS resolution depends on DoT-capable upstream resolvers being reachable. Configuration deployed via forward.conf at /var/ipfire/dns/forward.conf.

**ADR-0009-sysctl-hardening-append.md:**
- Context: Phase 6 hardens kernel parameters via sysctl. IPFire's /etc/sysctl.conf contains critical routing params (net.ipv4.ip_forward=1). Overwriting the file would break WAN routing. But appending is not idempotent — re-running the deploy script would duplicate params.
- Decision: Append hardening params to /etc/sysctl.conf (never overwrite). Use grep-based idempotency check before appending: `grep -q "net.ipv4.conf.all.send_redirects" /etc/sysctl.conf || cat sysctl-hardening.conf >> /etc/sysctl.conf`.
- Rationale: Preserves IPFire's own routing settings (ip_forward=1 is critical). Appending with dedup check is safe to re-run. CIS Level 1 params (send_redirects, accept_source_route, accept_redirects, rp_filter, tcp_syncookies) added without risk. `sysctl -p` is already idempotent for loading.
- Consequences: Rebuild script must use the grep-before-append pattern (not bare `cat >>`). Drift detection checks /etc/sysctl.conf hash — any Core Update that modifies this file will trigger a drift alert. The sysctl-hardening.conf file in the repo is the source of truth for which params to add.

**ADR-0010-ecdsa-only-wui-cert.md:**
- Context: IPFire's WUI runs on HTTPS at port 444. The default self-signed certificate needed documentation. Investigation revealed IPFire generates ECDSA-only certificates (no RSA option in its certificate generation).
- Decision: Document the ECDSA-only WUI certificate as-is. Do not attempt to replace it with a custom certificate. Certificate is NOT stored in the git repo.
- Rationale: IPFire's certificate generation is ECDSA-only — this is a platform constraint, not a choice. The self-signed cert is acceptable for a management interface on a trusted LAN. Storing certificates in git is a security anti-pattern (D-04). The certificate must be manually provisioned before rebuild (documented in rebuild prerequisites).
- Consequences: WUI cert documented in docs/wui-certificate.md for reference. Rebuild script Section 0 prerequisites list cert as a manual provisioning step. No automated cert deployment — IPFire's WUI cert is generated on first boot.

**ADR-0011-modular-validation-suite.md:**
- Context: Need to validate that all 6 phases of IPFire configuration are correct. A single monolithic validation script would be unwieldy and difficult to maintain as new phases are added.
- Decision: Per-phase validation scripts (validate-phase1.sh through validate-phase6.sh) orchestrated by a single validate-all.sh runner. Each script exits 0 (pass) or 1 (fail). validate-all.sh calls each in order and aggregates results.
- Rationale: Modular design allows running individual phase validation in isolation (--phase N flag). Each script is independently testable. Adding a new phase means adding one script, not modifying a monolith. Pass/fail/skip pattern is simple and consistent across all scripts. Phase 5 validation runs remotely via SSH to the monitoring host (different execution model, cleanly encapsulated).
- Consequences: validate-all.sh is the single acceptance gate — rebuild is not complete until it passes (D-06). Phase 5 requires SSH from IPFire to supportTAK-server (192.168.1.101). SKIP status is valid (e.g., supportTAK-server offline = SKIP, not FAIL).

**ADR-0012-git-rebuild-as-ha-strategy.md:**
- Context: The appliance needs a recovery strategy for hardware failure or corruption. Traditional HA approaches include LVM/ZFS snapshots, hot standby, or clustering. IPFire runs on a minimal Linux installation without LVM or ZFS.
- Decision: Git-based rebuild is the HA strategy. All configs, scripts, and manifests live in a git repo. A rebuild script applies the full configuration to a fresh IPFire install idempotently. Target RTO: 15 minutes.
- Rationale: IPFire's minimal environment has no LVM/ZFS snapshot support. Hot standby/clustering is explicitly out of scope (adds operational complexity disproportionate to SOHO scale). Git provides full version history of every config change. Rebuild script is testable and repeatable. SCP-based deployment (not git clone on IPFire) keeps IPFire tool-free.
- Consequences: Hardware failure requires: (1) install fresh IPFire on replacement hardware, (2) provision SSH key and WUI cert, (3) run rebuild.sh from dev machine. IPFire does not have git installed — the repo lives on the dev machine only. All config changes must be committed to git to be reproducible. WUI-managed settings require manual recreation (documented in rebuild script output).
