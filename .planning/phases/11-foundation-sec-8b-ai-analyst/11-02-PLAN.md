All commands run via SSH to opsadmin@192.168.1.22.

STEP 1 — Capture Malcolm-only RAM baseline (model should be unloaded from idle):
```bash
ssh opsadmin@192.168.1.22 "ollama ps"
# Should show no loaded models (idle unload from KEEP_ALIVE=5m)
# If model is loaded, wait for unload or run: ollama stop hf.co/fdtn-ai/Foundation-Sec-8B-Q4_K_M-GGUF
ssh opsadmin@192.168.1.22 "free -m"
```
Record the "used" value as Malcolm-only baseline.

STEP 2 — Run verbose inference benchmark (3 different prompts for variance):

Prompt 1 — Short security question:
```bash
ssh opsadmin@192.168.1.22 "ollama run hf.co/fdtn-ai/Foundation-Sec-8B-Q4_K_M-GGUF --verbose 'What is CVE-2024-3094 (xz backdoor)? Summarize the attack vector and recommend detection steps.' 2>&1 | tail -10"
```

Prompt 2 — ATT&CK mapping task (typical triage workload):
```bash
ssh opsadmin@192.168.1.22 "ollama run hf.co/fdtn-ai/Foundation-Sec-8B-Q4_K_M-GGUF --verbose 'A Suricata alert shows ET MALWARE Win32/Emotet Activity (POST) from 192.168.1.50 to 203.0.113.42:8080. Map this to MITRE ATT&CK techniques and recommend investigation steps.' 2>&1 | tail -10"
```

Prompt 3 — Longer analysis (stress test):
```bash
ssh opsadmin@192.168.1.22 "ollama run hf.co/fdtn-ai/Foundation-Sec-8B-Q4_K_M-GGUF --verbose 'You are a security analyst reviewing network logs. Explain the differences between lateral movement techniques T1021.001 (Remote Desktop Protocol), T1021.002 (SMB/Windows Admin Shares), and T1021.004 (SSH). For each, describe detection signatures a Suricata IDS could generate and recommend specific rule categories from the Emerging Threats ruleset.' 2>&1 | tail -10"
```

For each prompt, extract from the `--verbose` output:
- `prompt eval rate: X.XX tokens/s` (pp — prompt processing speed)
- `eval rate: X.XX tokens/s` (tg — token generation speed, THIS IS THE KEY METRIC)
- `eval count: NNN token(s)` (response length)

STEP 3 — Measure RAM with model loaded (immediately after inference, model still in RAM):
```bash
ssh opsadmin@192.168.1.22 "free -m"
ssh opsadmin@192.168.1.22 "ollama ps"
ssh opsadmin@192.168.1.22 "docker stats --no-stream --format 'table {{.Name}}\t{{.MemUsage}}' | head -10"
```
Record total used RAM. This is the "simultaneous load" measurement.

Check for OOM events:
```bash
ssh opsadmin@192.168.1.22 "sudo dmesg | grep -i oom | tail -5 || echo 'No OOM events'"
```

STEP 4 — Verify model unload after idle:
Wait 6 minutes (OLLAMA_KEEP_ALIVE=5m + 1 minute buffer):
```bash
# Note: This is a long wait. Run the sleep on the remote side.
ssh opsadmin@192.168.1.22 "echo 'Waiting 360 seconds for model unload...' && sleep 360 && ollama ps && free -m"
```
`ollama ps` should show no loaded models. `free -m` should show RAM recovered to near Malcolm-only baseline.

STEP 5 — Evaluate Phase 13 gate:
Compare measured tg (token generation) rate against the 2 t/s threshold from STATE.md:
- If tg >= 2 t/s: Phase 13 can proceed with standard batch triage design
- If tg < 2 t/s: Phase 13 requires design revision (document this clearly)

STEP 6 — If RAM exceeds 15.5 GB during simultaneous load:
Add OLLAMA_CONTEXT_LENGTH=2048 to the override and re-measure:
```bash
ssh opsadmin@192.168.1.22 "sudo tee /etc/systemd/system/ollama.service.d/override.conf > /dev/null << 'OVERRIDE'
[Service]
Environment=\"OLLAMA_HOST=127.0.0.1\"
Environment=\"OLLAMA_KEEP_ALIVE=5m\"
Environment=\"OLLAMA_CONTEXT_LENGTH=2048\"
OVERRIDE"
ssh opsadmin@192.168.1.22 "sudo systemctl daemon-reload && sudo systemctl restart ollama"
```
Then re-run one benchmark prompt and re-measure RAM.

STEP 7 — Create benchmark document at `docs/benchmarks/n150-foundation-sec-8b-benchmark.md`:

The document MUST contain:
- Hardware: Intel N150, 4 cores, 16GB DDR5 (single-channel), Ubuntu 22.04
- Model: Foundation-Sec-8B Q4_K_M (4.92 GB) via Ollama
- Date of benchmark
- For each prompt: prompt eval rate (pp t/s), eval rate (tg t/s), eval count (tokens), total duration
- Average tg t/s across all 3 prompts
- RAM measurements: Malcolm-only baseline, simultaneous load, post-unload recovery
- Whether OLLAMA_CONTEXT_LENGTH=2048 was needed
- OOM check result
- Phase 13 gate evaluation: tg rate vs 2 t/s threshold
- Raw verbose output appended as an appendix

Format as Markdown with tables for the benchmark results.
