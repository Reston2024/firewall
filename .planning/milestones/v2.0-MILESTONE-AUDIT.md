## Partial Closure: TRI-06

TRI-06 requirement text: *"End-to-end verified: IPFire alert → Malcolm
ingest → SOC pull → detect → investigate → recommend → receipt"*.

**What the Firewall side delivered (100% complete):**

- `contracts/execution-receipt.schema.json` v1.0.0 — authoritative 9-field schema
- `contracts/executor-gate.md` — ADR-E01 6-gate sequence documented
- `docs/tri06-receipt-contract.md` — one-page "railroad meeting point" contract
- `scripts/validate-tri06.sh` — 5-check E2E gate with `match`-query robustness and smoketest exclusion
- `scripts/emit-receipt.sh` — reference emitter for desktop forwarder authors
- `examples/receipts/*.json` — 5 reference receipts (one per `failure_taxonomy`)
- Live smoke test: schema-conformant receipt written to
  `triage-results-smoketest-20260410` via curl, read back via `match`
  query, validate-tri06.sh exclusion logic verified, index deleted
- Live reference-emitter test: fresh UUID v4 substituted into
  `examples/receipts/applied.json`, piped through
  `scripts/emit-receipt.sh`, wrote to `triage-results-2026.04.11`,
  `validate-tri06.sh` reported **TRI-06.4 PASS** with the real
  `receipt_id`, taxonomy, and `recommendation_id` echoed, then the
  test index was deleted so the gate returned to its natural SKIP
  state (no hand-crafted receipts polluting the audit)

**What remains (tracked as v2.1):**

The desktop SOC (local-ai-soc) already produces triage decisions and
persists them to its local store (per user confirmation during Phase 4).
TRI-06 full closure requires the desktop repo to add a forwarder that:

1. Reads the next un-forwarded triage decision from its local store
2. Transforms it into the 9-field execution-receipt schema
3. POSTs it to `triage-results-YYYY.MM.DD/_doc` on supportTAK (or pipes
   through `scripts/emit-receipt.sh`)
4. Marks the source decision as forwarded so it doesn't replay

**Why not closed in this session:** the session scope was strictly
"Firewall repo + IPFire SSH + supportTAK SSH" per user directive. The
desktop SOC and local-ai-soc repo are out of scope ("you do not do
anything related to the desktop"). Forcing TRI-06 closure would have
required a hand-crafted test receipt masquerading as a real AI decision
— explicitly rejected during Phase 4 as "a false PASS" and deleted.

**Release gate implication:** CLAUDE.md requires
`scripts/validate-all.sh — 0 FAIL`. That's satisfied: TRI-06 is SKIP
not FAIL, SKIPs are explicitly allowed, and validate-all.sh returns
exit 0.

## Tech Debt

### Phase 13 — Alert Triage

- `triage-results-template` uses OpenSearch dynamic mapping for the 9
  execution-receipt fields. Strings auto-type as `text` with `.keyword`
  subfields, ISO-8601 as `date`. Works correctly; explicit typed
  mapping would be cleaner. **v2.1 refinement.**

### Phase 14 — PCAP + Supply Chain

- **Cosign keyless requires interactive OAuth** for local runs. Move
  release signing to GitHub Actions with `id-token: write` for stable
  workflow OIDC identity. **v2.1 automation.**
- **68 High CVEs** in the Ubuntu 22.04 system SBOM need triage. Most
  are in EOL packages (nodejs 12.22) or "won't fix" upstream (libqt5webkit5).
  Ubuntu 24.04 upgrade is the most effective mitigation but is its own
  planning exercise (LUKS reboot risk, Malcolm recompat). **v2.1+ task.**
- **sbom-system.json scope** is `/var/lib/dpkg` only, not full rootfs.
  Docker image layer packages are out of v2.0 scope due to N150 16 GB
  RAM pressure. **Revisit after supportTAK RAM upgrade.**
- **IPFire → supportTAK SSH** is not provisioned (`Host key verification
  failed` observed). `scripts/validate-all.sh --host all` therefore
  cannot run from IPFire alone. **v2.1 key provisioning.** Workaround
  today: `scripts/validate-all.sh` auto-detects host role and runs the
  half of the suite each host can handle; the other half must be run
  from the other host.

### Orchestration discipline

- `scripts/validate-phase9.sh` had three stale checks (MAL-01c equality
  vs >=, MAL-01e grep matching user processes, MAL-06 assuming pre-SPAN
  world) exposed and fixed during Phase 3 orchestration work. All three
  were latent since ~2026-04-09 (SPAN acquisition) or earlier. Lesson:
  run the full validation suite whenever the world changes, not just
  during milestone closure.
